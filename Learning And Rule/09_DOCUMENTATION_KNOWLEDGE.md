# 📘 09 — Documentation & Knowledge Transfer Rules v1.1
**When to apply:** When documenting anything — features, processes, learnings, or final deliverables

---

## Rule 1: Step-by-Step Documentation
- Document the entire process step-by-step
- Each step gets its own section
- Flow: Planning → Design → Development → Testing → Deployment → Maintenance
- Each section clearly states: what work happens, what to care about, how to proceed

## Rule 2: Lessons Learned Document
- Maintain a "Lessons Learned" document
- Record: what worked in production, what didn't
- This becomes the reference point for every future update and new version
- New developers/teams can continue without losing past experience

## Rule 3: Step-wise Learning Structure
In documentation, for each step:
1. First: Things that DIDN'T work (so new team avoids same mistakes)
2. Then: Successful methods in order (best method first, then alternatives, then backups)
- Clear guidance at every step, no time wasted

## Rule 4: Decision Logging
Record all important decisions:
- WHY a feature was chosen
- WHY something was rejected
- What alternatives were considered
- This is a goldmine for future development

## Rule 5: Final User Documentation
When software is ready:
- Create comprehensive user documentation
- Cover: all features, complete flow, input settings, formats, examples
- Include: what to do, what NOT to do
- Design like a professional website with menus, search, screenshots

## Rule 6: Architecture Diagrams
Include in documentation:
- System Architecture Diagrams (High-Level + Low-Level)
- Data Flow Diagrams (DFD)
- Sequence Diagrams (important workflows)
- Component Interaction Maps

## Rule 7: Roles & Responsibility
- Divide project into different roles/modules
- Clearly define responsibilities for each
- Makes team collaboration easier

## Rule 8: Usability Over Complexity
- Focus on user convenience over complex security
- Remove unnecessarily complex methods
- Choose approaches that are fast, smooth, and stable

## Rule 9: Tool Creation Guidelines (Diamond Grade)
When converting completed work into a tool/software:

**Before Creating:**
- Analyze the full chat/session from start — carefully extract all learnings
- Extract: all failed approaches, all working approaches, best optimized logic, mistakes to avoid, useful code snippets

**Building the Tool:**
- Each task must be: step-by-step, no ambiguity, fully executable
- Use working scripts if available (for speed)
- Add fallback methods if primary fails
- Diamond grade = simple + stable + correct + complete

**Memory Handling:**
- Never rely only on chat history (AI may forget past instructions)
- Always store steps, logic, results in local files
- Previous working code = use as default for next run, speeds up process

**Task Building Flow:**
1. Ask questions to get proper notes
2. Save all answers in a file
3. Analyze: what worked, what failed
4. Build tool using ALL learnings
