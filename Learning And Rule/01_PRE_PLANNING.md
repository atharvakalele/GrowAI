# 🧠 01 — Pre-Planning & Research Rules v1.1
**When to apply:** Before starting ANY new task, feature, or project

---

## 🌟 Golden Rule: "Automate Business > Everything Else"
- Focus = BOOST & AUTOMATE business. Achieve goals faster & better.
- Don't divert on minor/useless features
- Backend = Nuclear Reactor ⚛️ (powerful, complex inside) | Frontend = Sweet Shop 🍬 (simple, easy outside)
- Architect fully, build minimum, scale on the go
- Divide features in phases, implement phase-wise, priority: must-have → advanced

---

## Rule 1: Complete A-to-Z Pre-Plan
Before writing a single line of code, plan everything:
- What are ALL the steps needed?
- What is the complete process flow?
- What inputs are required? What outputs are expected?
- List ALL features that need to be built
- **Only start after full plan is ready**
- Reason: Adding features mid-way breaks existing functionality

## Rule 2: Research Before Building
- Do thorough research before building anything
- **Never rely on trial-and-error** — it wastes time
- Only implement steps that have near 100% success probability
- If in doubt, **ask and discuss first**

## Rule 3: Official Documentation Priority
- **ALWAYS prioritize latest official docs** from the API/service provider
- API providers periodically change endpoints, functions, methods
- Old public information on the internet may be outdated
- Check official changelog/migration guides before implementing

## Rule 4: Version Compatibility Check
- Every programming language and API has multiple versions
- AI models are trained on internet data — high chance of mixing up old/new methods
- **Before adding any feature:** confirm which API version, language version we are using
- Only use code/functions compatible with OUR specific version
- Never accidentally use deprecated methods from older versions

## Rule 5: Feasibility Check
- Before building or editing any feature:
  - Validate feasibility using docs, internet, AI knowledge
  - Confirm the feature is realistically possible with our current tech stack
- This prevents wasted effort on non-viable ideas

## Rule 6: Feature Scope Definition
For large features, clearly define:
- Input fields and their default values
- Complete step-by-step flow
- Data sources and expected outputs
- All dependencies (UI, scripts, APIs, etc.)

If feature is too complex:
- Break into smaller parts
- Build in separate phases/sessions
- This reduces errors and makes debugging manageable

## Rule 7: Expert Recommendations
- As a software expert, give strong recommendations:
  - What SHOULD be done
  - What should NOT be done
  - Which additional features would benefit long-term
- Goal: Avoid repeatedly adding new features later

---

## 🔴 Mistakes to Avoid (From Our Experience)
1. ❌ Not checking official Amazon docs → used wrong SP-API endpoint (FE instead of EU)
2. ❌ Assuming India = Far East region → India is actually in EUROPE selling region
3. ❌ Starting implementation without verifying endpoint → wasted full day
4. ❌ Relying on general knowledge instead of checking latest docs
